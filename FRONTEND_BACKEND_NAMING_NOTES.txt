CATATAN PENAMAAN DATA FRONTEND <-> BACKEND
Project: CekTenang / Student Stress Detector

Tujuan file ini:
Menyamakan nama data yang dipakai frontend dengan kontrak backend supaya halaman lain
tidak bingung antara nama field untuk request dan response.


ATURAN UMUM
1. Request body dari frontend ke backend pakai camelCase.
   Contoh: sleepHours, activityDate, refreshToken, targetEmail.

2. Response data yang berasal langsung dari database biasanya balik dalam snake_case.
   Contoh: sleep_hours, activity_date, stress_score, created_at.

3. Response wrapper backend selalu seperti ini:
   {
     code,
     status,
     message,
     data
   }

4. Endpoint yang butuh login wajib kirim header:
   Authorization: Bearer <accessToken>

5. Token frontend disimpan di localStorage:
   accessToken
   refreshToken

6. Base URL frontend:
   import.meta.env.VITE_API_BASE_URL || "http://localhost:5000"

SARAN POLA DI FRONTEND
1. Untuk data form yang dikirim ke backend, ikuti nama camelCase backend.
   Contoh state form activity:
   {
     activityDate,
     sleepHours,
     moodScore
   }

2. Untuk data response, boleh langsung baca snake_case dari backend.
   Contoh:
   activity.sleep_hours
   prediction.stress_score
   recommendation.recommendation_text

3. Kalau ingin UI lebih rapi, mapping response snake_case ke camelCase boleh dilakukan
   di file service, bukan di komponen page.

4. Hindari bikin nama field baru di page tanpa mapping jelas.
   Contoh yang aman:
   input name="sleepHours" -> backend sleepHours

   Contoh yang perlu mapping:
   input name="sleep" -> harus diubah menjadi sleepHours sebelum POST.
